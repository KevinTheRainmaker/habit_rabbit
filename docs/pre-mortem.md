## Paper Tigers

_무섭게 느껴지지만 실제로는 관리 가능한 위험._

### PT1: "한국 사용자는 구독 모델을 안 쓴다"

**왜 Paper Tiger인가**: 틀렸다. 토스(Toss Pro), 클래스101, 밀리의서재 등이 한국 구독 시장을 이미 정착시켰다. 문제는 구독 모델 자체가 아니라 "구독할 만한 가치를 못 느끼는" 것이다. 이건 프리모텀의 다른 리스크들로 이미 다루고 있다.

**Paper Tiger에서 진짜 Tiger가 되는 조건**: 경쟁 무료 앱이 유사 기능을 무료로 제공할 때.

---

### PT2: "토끼 컨셉이 유치해서 성인 사용자가 안 쓸 것이다"

**왜 Paper Tiger인가**: Duolingo(올빼미), Headspace(캐릭터), Finch(자기돌봄 새 앱)가 이미 성인 대상 캐릭터 기반 앱의 성공을 증명했다. 한국에서도 카카오프렌즈 등 캐릭터 애착이 강한 문화적 맥락이 있다.

**Paper Tiger에서 진짜 Tiger가 되는 조건**: 앱스토어 스크린샷이 캐릭터보다 "할 일 목록"처럼 보일 때. 첫인상 디자인 품질이 관건.

---

### PT3: "App Store 심사가 너무 오래 걸려서 출시가 늦어진다"

**왜 Paper Tiger인가**: 일반 앱 심사는 24-72시간. 거절 이유는 대부분 예측 가능하고 사전 방지 가능하다 (Apple Sign-In T2, 개인정보처리방침 포함 여부, 구독 UI 요구사항).

**주의**: T2 (Apple Sign-In 미구현)가 해결되지 않으면 진짜 Tiger로 전환.

---

### PT4: "RevenueCat이 서비스 장애가 나면 구독이 안 된다"

**왜 Paper Tiger인가**: RevenueCat SLA 99.9%. 장애 시에도 로컬 캐시된 entitlement로 기존 유료 사용자는 기능 유지 가능 (TC-046 설계). 신규 구독만 일시 불가 — 치명적이지 않음.

**Paper Tiger에서 진짜 Tiger가 되는 조건**: 캐시 오프라인 정책 미구현 시.

---

## Elephants in the Room

_팀이 알지만 말하기 불편한 리스크. 반드시 직면해야 함._

### 🐘 Elephant 1: 아이템 경제 시스템이 1인 MVP 내에 완성 불가능할 수 있다

**불편한 진실**: 당근 포인트 적립(S9) + 아이템 샵(S10) + 토끼/굴 꾸미기(S11) + 미션(S12) + 프리미엄 아이템 게이팅 + 당근 경제 밸런싱 + 아이템 디자인 에셋 15-20개 제작... 이 전체를 1인이 MVP에서 완성하면서 동시에 RevenueCat 연동, 스트릭 시스템, AI 코칭까지 구현하는 것은 **현실적으로 6개월 이상 소요**될 수 있다.

**핵심 위험**: "아이템 샵은 나중에"라고 결정하면 당근 포인트를 적립해도 쓸 곳이 없어 핵심 루프가 붕괴된다. 아이템 샵 없는 당근 포인트는 의미 없다.

**대화 시작점**: "Phase 1에서 아이템 샵을 빼면 핵심 루프가 작동하는가? 당근 포인트만 있고 쓸 곳이 없는 베타를 출시할 수 있는가?"

**권고**: E1 사전 등록 결과 전까지 아이템 샵/꾸미기는 개발 시작 금지. 수요 증명 후에만 투자. 혹은 MVP에서 아이템을 3-5개로 극단적으로 줄이고 디자인 에셋을 외주 처리.

---

### 🐘 Elephant 2: "1인 개발자 번아웃"이 가장 현실적인 실패 원인이다

**불편한 진실**: 기술 리스크, 시장 리스크보다 더 높은 확률로 이 프로젝트를 중단시킬 요인은 개발자 본인의 번아웃이다. 28개 유저 스토리, 52개 테스트 시나리오, RevenueCat 연동, 아이템 디자인, AI 코칭 API, 마케팅, 고객 지원을 모두 혼자 한다. 어느 시점에 "이게 맞나?"라는 의문이 든다.

**실패 패턴**: 개발 3-4개월 차에 기능 완성도 70%, 아이템 디자인 미완, 테스트 부족 상태로 "일단 출시"하거나 무기한 연기.

**대화 시작점**: "6개월 후 내가 이 프로젝트를 계속하고 있을 조건은 무엇인가? 중간 마일스톤(베타 10명, 사전 등록 200명)이 실패하면 어떻게 할 것인가?"

**권고**: Phase 0 (랜딩 페이지 2주)를 반드시 먼저 진행. 수요 신호 없이 풀 개발 진입 금지. 중간 지점마다 "계속할 것인가" 결정 기준을 미리 명문화.

---

### 🐘 Elephant 3: 습관 3개 무료 한도가 실제로는 너무 빡빡할 수 있다

**불편한 진실**: Reddit 리서치에서 사용자들이 가장 싫어하는 것 중 하나가 "가치 전 업셀"이다. 3개 한도는 새 습관을 추가할 때마다 느끼는 즉각적인 마찰이다. Habitica는 무제한 무료다. Streaks는 기본 6개다.

"3개면 충분하다"는 가정이 검증된 적 없다. 사용자가 "3개밖에 안 되네, 삭제"라고 반응할 경우, 첫 실패 위로 플로우나 복구권 같은 정교한 설계가 의미 없어진다.

**대화 시작점**: "3개 한도를 선택한 이유가 수익화 압박인가, 아니면 사용자 경험 근거가 있는가? 5개로 늘리면 전환율이 떨어지는가, 오히려 리텐션이 높아지는가?"

**권고**: 베타에서 A/B 테스트 — 무료 한도 3개 그룹 vs 5개 그룹으로 D30 리텐션과 전환율 동시 비교. 결정을 데이터로.

## Go/No-Go 체크리스트

### 출시 전 필수 (런칭 블로킹 Tigers)

- [ ] **T1**: RevenueCat 결제→기능 활성화 E2E 플로우 베타 검증 완료
- [ ] **T2**: Apple Sign-In 구현 및 App Store 심사 요구사항 사전 검토 완료
- [ ] **T3**: 당근 포인트 중복 지급 방지 서버 멱등성 구현 + TC-020 통과
- [ ] 아이템 최소 15개 이상 준비 (T4 방지)
- [ ] TC-044 (자정 경계), TC-048 (타임존) 테스트 통과
- [ ] 개인정보처리방침 + 이용약관 앱 내 링크 포함

### 출시 후 첫 2주 내 모니터링 시작

- [ ] RevenueCat 대시보드 전환율 주간 체크 (T9)
- [ ] 사용자당 AI API 비용 추적 시작 (T8)
- [ ] 당근 적립/지출 패턴 분석 (T10)
- [ ] 앱스토어 리뷰 일일 모니터링 (T5, T6 조기 감지)

### 출시 전 결정 (Elephant 대응)

- [ ] E1 사전 등록 결과 확인 후에만 아이템 샵 개발 시작 결정 (Elephant 1)
- [ ] 무료 한도 3개 vs 5개 베타 A/B 테스트 계획 수립 (Elephant 3)
- [ ] Phase별 "계속/중단" 기준 명문화 (Elephant 2)

### 롤백 계획

- **구독 오류 발생 시**: RevenueCat 대시보드에서 수동 구독 복원 즉시 가능. 사용자 DM 대응 채널 준비
- **당근 중복 지급 발생 시**: 원격 설정으로 당근 지급 기능 일시 비활성화 후 수동 보정
- **아이템 샵 심각 버그 시**: 샵 화면 비활성화 (당근 포인트는 유지), 다음 업데이트에서 수정

## 최종 권고

**출시 가능 조건**: T1, T2, T3 해결 + 아이템 15개 이상 + Elephant 2 (Phase 0 검증) 통과

**가장 위험한 시나리오**: Elephant 1 (아이템 경제 미완성) + T3 (포인트 중복) + T9 (전환율 미달)이 동시에 발생하는 경우. "당근 쌓이는데 쓸 곳 없고, 포인트는 두 번씩 들어오고, 아무도 돈 안 내는" 상황.

**가장 높은 가치의 리스크 대응**: Elephant 3 해결 (무료 한도 A/B 테스트). 단순한 숫자 결정이 전환율과 리텐션에 가장 큰 영향을 미칠 수 있다.
